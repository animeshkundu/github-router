# github-router

A reverse proxy that exposes GitHub Copilot as OpenAI and Anthropic compatible API endpoints.

## Primary deployment target

The primary deployment target for this project is **Windows 11**. macOS and Linux are supported and tested, but Windows is the canonical user environment — every PR must pass the `windows-latest` CI job before merge. A Windows CI failure is treated as a merge blocker, not as flake. If Windows behavior diverges from POSIX, the Windows path is the authoritative one to fix; do not POSIX-skip a Windows failure to land a change.

The canonical end-user setup is **Windows 11 + Claude Code (the CLI client) + a GitHub Copilot _Enterprise_ license**. The enterprise license is load-bearing for feature decisions: it unlocks catalog models gated `restricted_to: ["enterprise"]` and the 1M-context Opus variants — `claude-opus-4.8` (the spawned-Claude-Code default; the single base slug already advertises `max_context_window_tokens: 1_000_000` — no `-1m` sibling needed), `claude-opus-4.7-1m-internal` (stand_in's Opus row), and `claude-opus-4.6-1m` (opus_critic's pinned model). When choosing models or windows, assume the enterprise catalog is present (prefer the 1M variants), but always gate on the live catalog with a non-enterprise fallback so the proxy still works on lesser tiers.

## Review checklist (read before submitting / approving any PR)

- **Stream lifecycle**: every `controller.enqueue` / `controller.close` / `reader.read` call site must have a regression test that intentionally races consumer cancel against the call. Cooperative-mock tests are insufficient — they cannot reproduce the microsecond window where Bun's HTTP layer closes the controller while a `pull()` is mid-`await`. See `tests/integration/chaos.test.ts` for the test pattern.
- **The smoking gun**: a new `Could not deliver error event` warn-log is a bug, not a routine warning. Open an issue and treat as a regression.
- **Author responsibility**: PR descriptions must list the failure modes the author considered and tested, not just the happy path. Reviewers can only check what they're asked about — narrow prompts produce narrow reviews. The class of bug we missed in the manual `ReadableStream({pull})` rollout was an enqueue-after-cancel race that the catch block was clearly intended to handle, but no test ever reproduced it. The catch-handler's existence is not a substitute for an actual race-triggering test.
- **Spec ≠ runtime**: WHATWG/Anthropic spec compliance is necessary but not sufficient. Verify what Bun (and Node undici when relevant) actually throw at runtime. Don't reason "the spec says close is idempotent" — verify "Bun throws `TypeError: Invalid state: Controller is already closed` if you enqueue after close."
- **Bun request-signal quirk**: `c.req.raw.signal` from a Bun/srvx HTTP handler is aborted as soon as the request body is fully consumed (i.e., right after `await c.req.json()`), even when the consumer is still happily reading the response. Do NOT propagate it into upstream `fetch()` calls — every such call would fail immediately with "This operation was aborted." `/v1/responses` and `/mcp` both intentionally drop it; tear-down on consumer cancel is handled at the `ReadableStream.cancel()` callback for streaming responses, and is a no-op for non-streaming responses (the upstream call completes regardless). If a future change truly needs to propagate consumer cancel, verify with a real Bun.serve listener — unit tests with `app.request(new Request(...))` do not reproduce the quirk.
- **Compatibility probe rule**: every field, header, body shape, or tool type that any client (Claude Code, Codex, raw API users) emits MUST have a probe row in `scripts/probe-copilot-compat.sh` AND a row in `docs/copilot-compat-matrix.md` — with an explicit accept-or-reject expectation. Discovery sources: real traffic (`bun run discover:fields` after launching with `GH_ROUTER_LOG_FIELDS=1`), code changes that emit new shapes, exploratory probing. The probe set grows monotonically; removing a row requires written justification in the matrix doc. Run `bun run probe:copilot` (strict mode) before merging changes that touch request shaping. Symmetric: both `❌ 400` and `✅ 200` rows are asserted, so drift in either direction surfaces immediately rather than after users hit it.
- **Strip-rule probe rule**: adding (or removing) a strip rule in `stripAnthropicOnlyFields` / `sanitizeCacheControl` / equivalent requires (a) an end-to-end probe in `scripts/probe-copilot-compat.sh` asserting the user-facing behavior the strip enables (typically a `200` where without the strip Copilot would 400), AND (b) a row in `docs/copilot-compat-matrix.md` documenting the upstream truth. The probe id should be referenced in the strip's code comment so a future contributor following a breadcrumb lands on the empirical evidence.
- **Windows-first CI**: `windows-latest` CI must be green before merge. Any test that skips on `process.platform === "win32"` requires an explicit written justification (e.g., the symlink-confused-deputy probe at `tests/lib-paths.test.ts:67` is justified because file-typed symlinks legitimately require admin/Developer Mode). Adding new `win32 return` guards to existing tests is a regression of CI coverage on the primary deployment target.
- **Zero-tolerance for flaky / failing CI**: every PR must have a fully green CI run across the entire matrix before merge — no exceptions, no "rerun until green," no "the failing test is unrelated to my change." Flakes and pre-existing failures are merge blockers for whoever is next to land a PR, regardless of who introduced them. If you encounter a flaky test, root-cause it and either fix the underlying race / resource leak / timing bug, OR delete the test with a written justification in the PR description (suppressing via `.skip` or retry-loops is not acceptable — those convert visible bugs into invisible ones). The cost of pausing to fix a flake is bounded; the cost of a green-rate that creeps below 100% compounds into "CI doesn't mean anything" and bugs ship.
- **MCP tool surface minimality**: every input/output field of every MCP tool (the personas under `PERSONAS_READ`/`PERSONAS_WRITE` and the utility tools under `NON_PERSONA_MCP_TOOLS` in `src/lib/peer-mcp-personas.ts`) must be required to call the tool, model-tunable in a way that improves outcomes, OR directly actionable feedback the model can act on next call. Echoed inputs, diagnostic-only fields, and non-actionable failure metadata cost the model's context for no return. When adding or extending a tool, write "what would the model do with this?" for each proposed field; if the answer is "nothing," cut it. See [`docs/peer-mcp-design.md`](docs/peer-mcp-design.md) "Design principle: ruthlessly minimal MCP tool surface" for the worked example and the per-field rationale.
- **Version bump per PR**: every PR must include a `package.json` patch bump (`npm version patch --no-git-tag-version`, or hand-edit the `"version"` field). The release workflow loops `npm version patch` until it finds an unpublished value, then publishes the unscoped `github-router` package; the post-publish push of the bumped `package.json` back to master runs under `continue-on-error: true` and silently fails without `secrets.PAT`, so the repo-side version drifts behind npm. Bumping per PR keeps `package.json` honest, which keeps `github-router --version` and the startup banner honest, which makes crash reports actionable. The single source of truth for the running version is `getPackageVersion()` in `src/lib/version.ts` — it reads `package.json` at runtime (not build time) because release.yml builds BEFORE the bump loop, so a build-time inline would always ship the pre-bump value.

## Subcommands

- `github-router start` — run the proxy on its own (port 8787). Use when bringing your own client.
- `github-router claude [-m <model>]` — launch Claude Code with the proxy auto-configured (synthetic credentials, peer-MCP injected, onboarding-skip fields written into the per-launch mirror).
- `github-router codex [...]` — launch Codex CLI against the proxy with `OPENAI_API_KEY` and `OPENAI_BASE_URL` pinned to the synthetic dummy and the proxy URL.
- `github-router auth` — re-run the GitHub device-code login (writes the PAT to `~/.local/share/github-router/github_token`).
- `github-router check-usage` — pretty-print Copilot quota / premium / chat / completions snapshots.
- `github-router models [<pattern>] [--json]` — pretty-print the live Copilot model catalog with capabilities (family, tokenizer, context/output limits, supports flags, reasoning efforts, supported endpoints, billing/restrictions). With a positional `<pattern>` arg, fuzzy-substring-filters (case-insensitive) on id / name / vendor / family. `--json` emits the raw catalog. Fetches live each invocation; requires a valid Copilot token (auto-refreshed via `setupCopilotToken`).
- `github-router debug [...]` — internal diagnostics; not for user-facing flows.

## Architecture

- **Stack**: TypeScript / Bun / Hono / SSE streaming
- **Import alias**: `~/` maps to `src/`
- **Token storage**: `~/.local/share/github-router/github_token`

### Two patterns

1. **Passthrough**: Forward directly to Copilot API:
   - `/v1/chat/completions`, `/v1/responses`, `/v1/embeddings`
   - Service: `src/services/copilot/create-*.ts` → handler → route

2. **Passthrough with sanitization**: Forward to Copilot, stripping unsupported fields:
   - `/v1/messages` (Anthropic) → strip `cache_control.scope`, filter beta headers → Copilot `/v1/messages?beta=true`
   - `/v1/messages/count_tokens` → same sanitization → Copilot `/v1/messages/count_tokens?beta=true`

### Beta header filtering & stealth-vs-leverage

`github-router claude` defaults to **leverage** (extended-betas ON; 20 prefixes forwarded, `EXPLICITLY_STRIPPED_BETA_PREFIXES` deny-list catches anything Copilot 400s on); `start`/`codex` default to **VS Code stealth** (3 prefixes only). Body-level `budget`, `output_config.schema`, `betas`, and per-tool `eager_input_streaming` are stripped on `/v1/messages` and `/v1/messages/count_tokens`. Full allowlist + strip-list + opt-out flags in [`docs/beta-headers.md`](docs/beta-headers.md).

### Experimental Claude Code features auto-enabled

`github-router claude` auto-enables five `CLAUDE_CODE_*` env-var feature gates via a presence-based guard (any user-set value is preserved — set to `0`/`false`/`off` to opt out per-feature): `CLAUDE_CODE_ENABLE_EXPERIMENTAL_ADVISOR_TOOL`, `CLAUDE_CODE_FORK_SUBAGENT`, `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`, `CLAUDE_CODE_ENABLE_FINE_GRAINED_TOOL_STREAMING`, `CLAUDE_CODE_ENABLE_TASKS`. **Note**: FORK_SUBAGENT silently no-ops in `claude --print` headless mode (`Z8()` precondition). **Not auto-enabled**: `CLAUDE_CODE_ENABLE_GATEWAY_MODEL_DISCOVERY` (slug-registry mismatch silently degrades to lowest-common-denominator fallback — see [`docs/claude-env-injection.md`](docs/claude-env-injection.md) before changing). Full table + rationale in [`docs/claude-env-injection.md`](docs/claude-env-injection.md).

### Update reliability (Windows-safe), proxy self-update, LLM toolbelt

Three startup behaviors share one Windows-safe exec helper (`src/lib/exec.ts`): `resolveExecutable` (absolute-path resolution against PATH honoring PATHEXT, **excluding cwd** — closes the planted-`npm.cmd` vector) and `buildExecInvocation` (real cmd.exe quoting — caret-escapes `& | < > ( ) ! ^`, refuses `%`; never the naive quote-if-space). On Windows `claude`/`npm` are `.cmd` shims that `execFile` cannot launch, so every child-process call here goes through this helper.

- **Claude Code auto-update** (`src/lib/claude-version-check.ts`): `github-router claude` updates Claude Code on launch via **`claude update`** (install-method-aware — native installer or npm — so it never creates a conflicting second install; not blocked by the `DISABLE_AUTOUPDATER` the proxy sets in the child env via `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1`). Builds too old to have `claude update` fall back to `npm install -g @anthropic-ai/claude-code@latest` with a visible warning (never silently strand). Throttled 1h (`last-update-check`); `--auto-update` (default) installs, `--no-auto-update` warns only, `--no-update-check` skips. Cross-process install lock (`src/lib/update-lock.ts`).
- **Proxy self-update** (`src/lib/self-update.ts`): default-ON, throttled (`last-self-update-check`). Probes npm for the **unscoped `github-router`** (the scoped `@animeshkundu/github-router` is GitHub-Packages-only); on a newer version spawns a **detached, post-exit updater** (waits for this PID to exit, then `npm install -g github-router@latest`) — the proxy is a running process, so in-place install would hit Windows file locks (EPERM/EBUSY) and can corrupt the install. Applies on next launch; no re-exec. Disable with `--no-self-update` or `GH_ROUTER_NO_SELF_UPDATE=1`. Wired into `start`/`claude`/`codex` after `setupAndServe`.
- **LLM toolbelt** (`src/lib/toolbelt/`): curated CLI tools (`rg`, `fd`, `jq`, `sd`, `ast-grep`/`sg`, `yq`, `scc`, `difft`/difftastic, `gron`) materialized into `~/.local/share/github-router/bin/` and prepended to the spawned agent's PATH so the model calls them as native. **Gap-fill**: only tools NOT already on the user's PATH are exposed (never shadows a pinned `jq`/Go-vs-Python `yq`). `rg` is materialized from the existing `@vscode/ripgrep` binary (no second download); the rest lazy-download from version-pinned GitHub releases with **hardcoded-in-source SHA256** verified before extraction (`src/lib/toolbelt/manifest.ts`; regenerate via `scripts/gen-toolbelt-manifest.ts`, which takes an optional `--only <csv>` filter to re-pin a subset). `tokei` is NOT bundled (its GitHub releases ship 0 prebuilt binaries since 2024; `scc` is the bundlable code-stats substitute) but stays in the worker `toolbelt` allowlist opportunistically — it runs if the user has it system-installed. Extraction is dependency-free (`extract.ts`) and **regular-files-only** (rejects symlink/hardlink/device archive entries). Materialization runs in the background (never blocks launch); the awareness one-liner is injected into the mirrored CLAUDE.md so the agent knows which tools are on PATH. Anti-shadow: the launcher resolves `claude`/`codex` to an absolute path so the toolbelt PATH can't shadow the CLI itself. Opt out with `GH_ROUTER_DISABLE_TOOLBELT=1`; skip individual tools with `GH_ROUTER_TOOLBELT_SKIP=jq,yq`. The toolbelt PATH also reaches the worker-agent bash env (casing-safe, no global `process.env` mutation), and all worker modes additionally get a `toolbelt` tool (itself read-only) that runs these CLIs with `shell:false` (see "worker tools" below).


### Unsupported features (Copilot can't serve)

Files API (`/v1/files/*`) → 404; ADVISOR (`advisor-tool-2026-03-01`) → Phase I server-side wiring (proxy injects `__anthropic_advisor`, dispatches to gpt-5.5 xhigh, streams `advisor_tool_result` back); `mcp_servers` non-empty → fail-fast 400; Bridge / CCR remote-session env stripped from spawned-child env. Full surface + opt-out paths in [`docs/unsupported-features.md`](docs/unsupported-features.md).

### `apiKeyHelper` and external credential scripts

The user's `settings.json` is mirror-copied into `CLAUDE_CONFIG_DIR` at startup so any `apiKeyHelper` / `awsCredentialExport` / `awsAuthRefresh` / `gcpAuthRefresh` defined there still fires inside the proxy session. The proxy supplies auth via the synthetic `claudeAiOauth` blob in `<CLAUDE_CONFIG_DIR>/.credentials.json`; if a user's helper mints an additional `x-api-key` header it's sent alongside our Bearer (Copilot ignores `x-api-key`). See [`docs/auth-isolation.md`](docs/auth-isolation.md).

**MCPs**: user-scope and local-scope MCPs are stored in `~/.claude.json`, which is MIRRORED (snapshot-copied at proxy startup). To register MCPs that persist across `github-router claude` launches, add them via plain `claude mcp add` (or edit `~/.claude.json` directly) *outside* a proxy session — the next launch's snapshot will pick them up. MCPs added inside a proxy session are session-scoped and lost when that session ends; this is by design (the mirror is one-way snapshot, not write-back, to keep the proxy's session state isolated from the user's real config). **Subagent MCP visibility**: the five scoped servers (`peers`, `search`, `workers`, `decide`, plus `browser` under `--browse`, and the `codex-cli` stdio entry when `--codex-cli` is in effect) are merged into the mirrored `.claude.json`'s `mcpServers` so Agent-tool subagents, forks, and agent-teams subprocesses inherit them. Project-scope `<workspace>/.mcp.json` is untouched — spawned `claude` inherits the parent's cwd and reads it directly. Subagent `.md` frontmatter intentionally omits `tools:` (omission inherits the parent's full toolset; adding `tools:` would *restrict*). On collision with a user-side entry of one of our bare keys, `resolveGroupKeysFromMirror` walks `gh-router-<group>`, `gh-router-<group>-2`, … to the first free name (threaded into both the entry and the persona `.md` routing string) rather than dropping the group — it never skips and never reuses a user-owned name. If the whole mirror injection fails (unwritable), the proxy falls back to ephemeral `--mcp-config` for the parent session only (subagents degraded — visible at startup in the `subagent-INVISIBLE` banner).

### Default models

`claude` → `ANTHROPIC_MODEL=claude-opus-4-8` (Anthropic-published dashed slug — Claude Code's `/model` UI is backed by a registry of Anthropic-published slugs; setting Copilot's dotted `claude-opus-4.8` slug falls back to "Opus 4" with a "Newer version available" hint instead of selecting "Opus 4.8 (1M context)"). On enterprise tiers, `pickClaudeDefault` upgrades the default to `claude-opus-4-8[1m]` — the literal-bracket suffix that cc-backup `src/utils/context.ts:35-40` recognizes to unlock 1M-context accounting locally; the proxy's `resolveModel` strips the bracket before talking to Copilot. **Dual-signal 1M detection** in `src/lib/port.ts:pickClaudeDefault` flips `[1m]` on when EITHER (1) a sibling-slug `opus-${family}-1m*` exists in the catalog (how 4.6 and 4.7 ship) OR (2) the base-slug catalog entry advertises `capabilities.limits.max_context_window_tokens >= 1_000_000` (how 4.8 ships — single slug, no `-1m` sibling). The proxy's `resolveModel` translates Anthropic slugs (with or without `[1m]`) to Copilot's dotted slug at request time. Fallback chain: `claude-opus-4-7` → `claude-opus-4-6` → `claude-opus-4-5`. `codex` → `gpt-5.5`. `ANTHROPIC_SMALL_FAST_MODEL` defaults to `claude-sonnet-4-6` (quality lift on background ops over Haiku; the `/model` picker's Haiku tier row stays `claude-haiku-4-5`); `/model` picker tier rows are seeded via `ANTHROPIC_DEFAULT_{SONNET,HAIKU,OPUS}_MODEL` (bare slugs — Copilot has no 1M backend for sonnet/haiku, and Anthropic-side `modelSupports1M` doesn't list haiku). Opt out of 1M locally with `CLAUDE_CODE_DISABLE_1M_CONTEXT=1` (HIPAA). Fallback chains fire only on the implicit-default path. Full rationale + slug-translation walkthrough in [`docs/default-models.md`](docs/default-models.md).

### Peer-model MCP integration (auto-invocation, effort, decomposition)

The `claude` subcommand auto-injects peer-model review tools as Claude Code subagents (`codex-critic` gpt-5.5, `codex-reviewer` gpt-5.3-codex, `gemini-critic` gemini-3.1-pro-preview, and `gemini-reviewer` gemini-3.1-pro-preview at highest reasoning — a second-lab line-level reviewer, registered alongside `gemini-critic` whenever a `gemini-3.x-pro` model is in the catalog) plus a `peer-review-coordinator` meta-subagent. Each persona accepts an `effort?: "low"|"medium"|"high"|"xhigh"` argument; per-persona `allowedEfforts` table, the SSE-streamed `/mcp` xhigh-default rationale, the latency-by-effort matrix, and the predictedTooLong pre-flight cap live in [`docs/peer-mcp-design.md`](docs/peer-mcp-design.md). Concurrency cap `MAX_INFLIGHT_TOOLS_CALL = 32` (single shared counter in `src/lib/mcp-inflight.ts`; both MCP-boundary persona calls in `src/routes/mcp/handler.ts` AND nested `peer_review`/`advisor` calls inside the worker tools acquire from the same slot — so a worker can't fan out unboundedly and starve operator traffic). **Decomposition is the load-bearing fix for Claude Code v2.1.113+'s MCP per-tool-call timeout regression ([#50289](https://github.com/anthropics/claude-code/issues/50289)); the `MCP_TIMEOUT=600000` env injection is harmless insurance.** Two non-negotiables when touching `src/routes/mcp/handler.ts`: (1) the `predictedTooLong` cap MUST fire before the `acquireInFlightSlot()` call (don't reorder — leaks concurrency slots on every reject); (2) `sweepStalePeerAgentMdFiles`'s regex requires the digit-PID prefix — relaxing it would delete user-authored files like `peer-reviewer.md`.

**Five intent-named MCP servers (server split).** The MCP surface is split into FIVE scoped servers, each a distinct `mcpServers` entry pointing at a path-scoped `/mcp/<group>` endpoint, so the server name signals the tool category to the model instead of burying everything under one opaque `gh-router-peers`: `peers` (the critics + `gemini_reviewer` when a `gemini-3.x-pro` model is served + `codex_implementer` in `--codex-cli`), `search` (`code`, `web`), `workers` (`explore`, `review`, `implement`), `browser` (the `browser_*` tools, renamed — only with `--browse`), and `decide` (`stand_in`). The model sees `mcp__search__code`, `mcp__browser__navigate`, `mcp__peers__codex_critic`, etc. The single source of truth is `GROUP_META` + the `group` field on each tool in `src/lib/peer-mcp-personas.ts`; `assertMcpToolSurfaceConsistent()` (called at server boot) pins per-group + cross-union name uniqueness. **Config keys are bare** (`peers`/`search`/…); on collision with a user-side `mcpServers` entry, `resolveGroupKeysFromMirror` (in `src/lib/codex-mcp-config.ts`) walks the numbered fallback sequence `gh-router-<group>`, `gh-router-<group>-2`, … until it finds a free name, and threads the RESOLVED key into BOTH the `mcpServers` entries AND the persona `.md` routing strings — so a user's own `browser`/`search` MCP never silently hijacks or drops one of ours. It **never skips and never returns a name the user owns**: every enabled group is guaranteed a proxy-owned key, so a capability is never dropped and the model is never routed at the user's same-named server. The route (`src/routes/mcp/route.ts`) accepts `POST /mcp/:group` (scoped) and `POST /mcp` (the full union, retained for BYO `start`/`codex` clients); the handler filters `tools/list`/`tools/call` by URL scope (wrong-scope call → -32601). **Tool renames are MCP-facing only**: `code_search`→`code`, `web_search`→`web`, `worker_explore`→`explore`, `worker_implement`→`implement`, `browser_*`→strip the prefix; persona names and `stand_in` are unchanged. For browser tools the rename is applied at the `NON_PERSONA_MCP_TOOLS` spread, leaving the wire `tool` string the installed MV3 extension dispatches on (`browser_navigate`, …) physically unchanged in `src/lib/browser-mcp/index.ts` — so **no extension reload is needed**.

The same `/mcp` surface also exposes two non-persona utility tools (group `search`) that all clients see: `web` (Copilot-backed web search, was `web_search`) and `code` (was `code_search`). **`code` is semantic-first:** its default `mode: "semantic"` ranks by MEANING via ColBERT (the former standalone `semantic_search` tool, folded in here) and transparently falls back to lexical BM25F when the per-workspace index isn't ready, labelling a 3-valued `source` (`semantic` | `lexical` | `lexical-fallback`) so a degrade is never silent; semantic-ready rows additionally carry a ColBERT `score`. The `mode` enum is `semantic|lexical|exact|regex|ast`; the shared router is `src/lib/unified-code-search.ts` (`runUnifiedCodeSearch`). The BM25F / ranking / `complete` / `scan` / `ast_pattern` behavior described below is the LEXICAL path (`lexical` = the former `ranked`, `exact` = the former `literal`), reached by the forced modes and by the semantic fallback. `code` runs ripgrep with BM25F ranking (Robertson, Zaragoza, Taylor 2004) over four code-aware fields (matched line, surrounding context, file path tokens, symbol-definition heuristic), then refines the symbol-context field with tree-sitter AST analysis on the top hits (`structural: "full"` → top 50, `structural: "topN"` → top 10) so identifier definitions outrank incidental string matches. Tree-sitter passes are wrapped in a 200ms wall-clock budget; on exhaustion remaining hits drop back to the regex heuristic for that one file. The MCP response includes one optional `notice: string` field, present iff an actionable degradation fired — either the structural budget exhausted, or the 256KB response-size cap truncated the result set; the message text tells the model what to retry (`structural: "topN"`, narrow query, lower limit). The default `limit` is 200 (raised from 20 for exhaustive enumeration), and `summary` is ON BY DEFAULT — each response carries an `outlines` field: a tree-sitter structural outline (symbols + line numbers — full tree, nested methods/functions included) of the distinct files in the result set (first 10, in result order, via `outlineFile` in `src/lib/tree-sitter-grammars.ts`), a compact map of where matches live that augments, never replaces, each hit's `snippet` (set `summary: false` to omit it; outlines are budget-fitted after results so they never breach the 256KB envelope). In the lexical path a per-file `--max-count` (`RANKED_MAX_PER_FILE`) plus a shoulder cut (drop hits below 50% of the top score) keep one match-dense file from exhausting the 200-hit global limit and keep the model from being flooded — but BOTH are default-only precision filters: passing `complete: true` disables them and returns the full ripgrep match set (reordered, capped only by `limit`) — the recall floor in [`docs/code-search-floor.md`](docs/code-search-floor.md) applies to that lexical path: in `complete` mode it drops nothing by its own heuristics (only the explicit `limit`), is a superset of naive `rg <identifier>` for single ASCII identifiers, and is `=` the ripgrep it runs (the doc is precise that it does NOT subsume ast-grep's multi-line patterns or a full-workspace tree-sitter scan). When the default prune hides matches the response `notice` says so and points at `complete: true`, so a miss is never silent; and a hit carries `role: "definition"` only when the structural pass AST-confirms it (absence is never a "usage" claim). Single-identifier queries in the lexical modes are auto-expanded across camelCase / snake_case / kebab-case / SCREAMING_SNAKE skeletons before being passed to ripgrep (`getUserName` also finds `get_user_name`). Workspace is any absolute path the proxy process can read — no allow-set, no secret-shape file denylist. The threat model is symmetric (Claude Code already has Read / Bash / Edit that reach the same paths), so gating one tool was inconsistency, not defense. Ripgrep itself comes from `@vscode/ripgrep` (per-platform binary via optionalDependencies) with system-PATH `rg` preferred when available. The MCP handler trims the rich internal response to `{file, line, snippet}` per hit (plus `role`/`endLine`/`name`/`score` where present) and a tiny envelope carrying the top-level `source` before stringifying — internal diagnostics (BM25F scores, field contributions, scanned-files, elapsed-ms, ranking-metadata block) are intentionally NOT forwarded; see the "Design principle: ruthlessly minimal MCP tool surface" section in [`docs/peer-mcp-design.md`](docs/peer-mcp-design.md) for the per-field rationale. Implementation: `src/lib/unified-code-search.ts` (router) + `src/lib/code-search.ts` (lexical backend) + `src/lib/colbert/` (semantic).

`/mcp` also exposes three **worker tools** (group `workers`: `explore` read-only, `review` read-only, `implement` read+write — `explore`/`implement` were `worker_explore`/`worker_implement`) that delegate scoped work to an **autonomous Pi-runtime worker subagent** (the vendored Pi runtime lives at `src/vendor/pi/`; the sync protocol is in [`docs/pi-vendor-sync.md`](docs/pi-vendor-sync.md)). **Per-mode model defaults** (caller `model` arg always wins; all resolved against the live catalog with thinking clamped to the model's `reasoning_effort` allowlist): read-only `explore`/`review` → `DEFAULT_MODEL` = `gemini-3.5-flash` at `high` (1M context); read+write `implement` → `IMPLEMENT_DEFAULT_MODEL` = `gpt-5.5` at `xhigh` (coding wants max reasoning); `browse` → `BROWSE_DEFAULT_MODEL` = `gpt-5.4-mini`. All three constants live in `src/lib/worker-agent/engine.ts` (single source of truth, re-imported by the handler and the gate — no parallel constant to drift). **`review` is a third worker mode** (`WorkerAgentOpts.mode`): the SAME read-only tool surface as `explore` (`buildWorkerTools` returns the explore array for both) but `systemPromptFor("review")` adds a one-line reviewer ROLE frame (verify correctness against the actual code, report findings with severity + `file:line`) — role framing, not prescriptive step-advice, so it keeps faith with the no-scaffolding prompt principle. Unlike the stateless `peers` critics, the `review` worker can navigate the repo to check surrounding context itself. **Worker toolset:** explore/review expose 9 read-only tools — `read`/`glob`/`grep`/`code_search` (semantic-first via the shared `runUnifiedCodeSearch`, so workers get ColBERT + transparent lexical fallback too)/`web_search`/`fetch_url` plus a read-only `toolbelt` tool (runs `rg`/`fd`/`sg`/`jq`/`yq`/`gron`/`scc`/`tokei`/`difft`/read-only `git` with `shell:false` — args are literal, write/exec flags and mutating git subcommands rejected), `advisor` (consult a stronger cross-lab model on a focused concern; it sees the recent transcript via the engine's `getMessages` closure over `agent.state.messages`) and `update_plan` (a stateful planning checklist re-surfaced each turn by the engine's `transformContext` hook so it survives compaction — `appendPlanReminder` in `engine.ts`); implement adds `edit`/`write`/`bash`/`codex_review` (13 total). Workers are TERMINAL — no sub-worker spawning — and `peer_review` is deliberately NOT wired in (peer critics aren't part of the worker surface; `advisor` is the consultation path). Implement mode is no longer forced agent-wide sequential: pure read/search batches run in parallel, while `edit`/`write`/`bash`/`codex_review`/`update_plan` each declare `executionMode:"sequential"` so Pi serializes any batch containing one of them (a write/stateful tool never runs concurrently). Dual gate: `workerToolsEnabled()` drops all worker tools from `tools/list` AND `tools/call` when `GH_ROUTER_DISABLE_WORKER_TOOLS=1` OR the explore default `gemini-3.5-flash` (`WORKER_DEFAULT_MODEL`, aliased from `DEFAULT_MODEL`) is missing from the Copilot catalog (or present but lacks `tool_calls`). Implement's `gpt-5.5` is NOT a gate input — if absent, `implement` errors helpfully at call time rather than the whole worker surface vanishing. The worker tools accept optional `workspace` (absolute path) — defaults to the proxy's launch cwd, override when the parent agent has multiple workspaces open; absolute-only enforcement at the MCP boundary in `runWorkerToolCall` so a typo doesn't silently resolve against `process.cwd()`. Budget caps (`Budget` in `src/lib/worker-agent/budget.ts`): max turns 500 (`GH_ROUTER_WORKER_MAX_TURNS`), max wall-clock 30min (`GH_ROUTER_WORKER_MAX_WALLCLOCK_MS`), max cumulative tool-output bytes 16 MiB (`GH_ROUTER_WORKER_MAX_TOOL_BYTES`), max tool calls 250 (`GH_ROUTER_WORKER_MAX_TOOL_CALLS`), max CONSECUTIVE identical tool calls 3 (`GH_ROUTER_WORKER_MAX_REPEATED_CALLS` — the duplicate-read / anti-loop guard; blocks the repeated call without halting the run), advisor transcript chars 720 000 (`GH_ROUTER_WORKER_ADVISOR_MAX_CHARS`, matches `ADVISOR_MAX_CONVERSATION_CHARS` in `src/services/advisor/advisor.ts`). **Context management** (all worker modes, model-agnostic): a per-run `ContextBudget` (`src/lib/worker-agent/context-budget.ts`) sized from the resolved model's catalog window drives three defenses that never drift — structural model-free compaction via Pi's `transformContext` hook (`compaction.ts`: byte-floor trigger, `structuredClone` before mutating, stub old tool-results/bash-output/tool-call-args oldest-first + current-turn truncation, never orphaning a `toolCall`↔`toolResult` pair), a dynamic per-result `afterToolCall` cap (`tool-output-cap.ts`, ~30% of the window so most pages fit one read), and a request-boundary backstop in `stream-fn.ts` (covers BOTH chat and `/responses`) that stops a predicted overflow with a visible diagnostic instead of an opaque upstream 4xx. **No token/cost accounting** — model-agnostic proxies hit the runaway-loop concern without per-model tokenizer duplication. `implement` accepts `worktree: boolean` for git-worktree isolation (per-call auto-clean `finally` + session-end SIGINT/SIGTERM sweep + boot-time PID+UUID-gated sweep — the digit-PID prefix is load-bearing, do NOT relax). Worker `bash` runs through a **strict env allowlist** (NOT denylist) with POSIX `bash -c` (no `.profile`/`.bashrc`) and `taskkill /T /F` on Windows; **all `GH_ROUTER_*`, `GITHUB_TOKEN`, `ANTHROPIC_AUTH_TOKEN`, `OPENAI_API_KEY`, `COPILOT_TOKEN` are dropped** from the spawned bash env. Worker file IO (`read`/`write`/`edit`) caps at 10 MiB matching `MAX_STDOUT_BYTES` in `src/lib/code-search.ts`. Path denylist for the worker's internal `read`/`glob`/`grep`/`code_search` tools (these keep their internal names — distinct from the MCP-facing rename): `.env*`, `*.pem`, `id_rsa*`, `id_ed25519*`, `.git/` interior, `.ssh/`, `.gnupg/`, `.npmrc`, `.netrc`. Full architecture + per-cap table in [`docs/peer-mcp-design.md`](docs/peer-mcp-design.md) "Worker tools".

`/mcp` also exposes the **`stand_in` tool** — a server-side, code-driven 3-lab consensus advisor for **decision tiebreak when the user is away** (`src/lib/stand-in.ts`). Polls gpt-5.5 xhigh + claude-opus-4-7 xhigh + gemini-3.1-pro-preview high across **blind round 1 → informed round 2 → abstain**, returns a ranked-choice verdict (`consensus` / `majority` / `no_consensus` / `need_more_info`). Scope-bound to **advisor, not decider**: recommends, never executes; dangerous actions (push, delete, drop, deploy) still require the user even with three-lab consensus. Gate (`capability: "stand_in"` → `standInToolEnabled()`): drops from `tools/list` AND fails `tools/call` with -32601 when any of the three required models is missing from the live catalog. Slot accounting: **one slot per `stand_in` invocation, NOT one per internal model call** — the MCP boundary acquires the slot; `dispatchModelCall` (extracted from `callPersona` so non-persona callers can reuse the per-endpoint dispatch) does NOT re-acquire. A single `stand_in` call making 6 internal upstream fetches consumes exactly 1 slot from the cap=32 budget. `predictedTooLong` cap = 6KB on `decision + options + context` (JSON-path only; SSE bypasses) — fires BEFORE slot acquisition per the load-bearing invariant. **Three non-negotiables when touching the orchestrator**: (1) the protocol is code-driven, NOT model-driven — the abstain invariant and blind-R1 anti-sycophancy property must hold deterministically, not "if an orchestrator model honors them"; (2) `isError` stays `false` for `no_consensus` / `need_more_info` — those are valid protocol outcomes, not errors (`isError: true` is reserved for input-shape failures); (3) per-model effort is FIXED (not caller-tunable) — exposing knobs would invite callers to cheap out and muddy the consensus signal. Full design + verdict semantics + sibling-tool distinctions in [`docs/peer-mcp-design.md`](docs/peer-mcp-design.md) "stand_in tool (away-mode advisor)".

### Spawned-CLI auth isolation

`STRIPPED_PARENT_ENV_KEYS` (`src/lib/launch.ts`) sanitizes auth-related keys from the parent env BEFORE the proxy's overrides are merged in — prevents shell-exported credential leaks AND silences Claude Code's `Auth conflict` warnings. `getClaudeCodeEnvVars` sets `CLAUDE_CONFIG_DIR` to a router-owned mirror dir; `ensureClaudeConfigMirror` (`src/lib/paths.ts`) classifies each `~/.claude/` entry as **ISOLATED** (skipped — `.credentials.json`, lock files, `statsig/`, `cache/`, `paste-cache/`), **SHARED** (directory symlink, atomic temp+rename — `projects/`, `sessions/`, `tasks/`, `todos/`, `transcripts/`, `shell-snapshots/`, `plans/`, `file-history/`, `backups/`), or **MIRRORED** (snapshot copy — everything else). Two non-negotiable rules: **never symlink individual files** (Node's `fs.rename()` doesn't follow symlinks; Claude Code's atomic-write pattern severs them — gemini-critic finding) and **`agents/` MUST stay MIRRORED** (the proxy writes per-launch `peer-<pid>-<rand>-<name>.md` files there; a symlink would route writes/deletes into the user's real `~/.claude/agents/` and destroy custom subagent files — pinned by a `policyFor("agents") === "MIRRORED"` regression test). The proxy then writes a synthetic `claudeAiOauth` credential to the mirror so spawned teammates (whose env-allowlist drops `ANTHROPIC_AUTH_TOKEN`) can authenticate by reading the file, AND merges `hasCompletedOnboarding: true` + `bypassPermissionsModeAccepted: true` + a synthetic `oauthAccount` into the mirrored `.claude.json` (`injectSyntheticClaudeJsonFields`) — without this, fresh machines with no prior Claude Code state hit Claude Code's first-launch wizard including the "Sign in to your Anthropic account" OAuth step, which silently defeats the synthetic credential. **Source priority for the merge** is the canonical `~/.claude.json` HOME-level path (per binary resolver `qG`: `path.join(process.env.CLAUDE_CONFIG_DIR || homedir(), ".claude.json")`) → mirror file → `~/.claude/.claude.json` subdir-level (legacy) — the mirror walk can never see the HOME-level file, so this priority is what preserves user-scope MCPs added via `claude mcp add` on default-config users. An `assertOnboardingGateInjected` postcondition re-reads the mirror file and throws if the gate fields didn't land, converting every "warn and return" branch in the helper into a launch-failing error so the OAuth wizard cannot silently re-emerge. **Do NOT re-introduce `ANTHROPIC_AUTH_TOKEN=<anything>` to the spawned-child env — historical agent-teams silent-mailbox bug; teammates drop the token from spawn allowlist and land at "Not logged in".** Full mirror policy, synthetic-credential schema (verbatim from binary `guH`), `.claude.json` onboarding-gate field rationale + source-priority logic, keychain isolation, no-401 invariant (`forwardError` 401→503 remap), and migration-from-older-versions warning in [`docs/auth-isolation.md`](docs/auth-isolation.md).

### Browser-control MCP (`--browse`)

`github-router start --browse` (or `claude --browse` / `codex --browse`, or `GH_ROUTER_ENABLE_BROWSE=1`) opt-in flag adds 19 browser-control tools under the `browser` MCP server (`mcp__browser__*`) to the `/mcp` surface, letting the model drive a real Chrome / Edge browser through an MV3 extension shipped under `src/browser-ext/`. The capability gate `browserToolsEnabled()` (in `src/lib/mcp-capabilities.ts`, also consulted by `claude.ts` to decide whether to register the `browser` server) requires BOTH the opt-in AND a positive `hasSupportedBrowserInstalled()` check, mirroring the symmetric list-time + call-time pattern used by `workerToolsEnabled()` and `standInToolEnabled()`. MCP-facing tool names (prefix dropped): `list_tabs / open_tab / close_tab / navigate / read_page / scroll / screenshot / keyboard / wait / eval_js / download / mouse / drag / type / diagnostics / find / act / observe / extract`. **The rename is MCP-facing only** — the wire `tool` string the extension dispatches on stays `browser_*` (unchanged in `src/lib/browser-mcp/index.ts`), so installed extensions need no reload. Element refs returned by `read_page` are the primary input to subsequent act / mouse / drag calls — preferred over CSS selectors because refs survive dynamic class names. The humanlike-input set (`mouse / drag / type`) routes through CDP `Input.dispatchMouseEvent` / `dispatchKeyEvent` for trusted events, hover-to-reveal menus, drag-and-drop (auto-detects HTML5-DnD via `Input.setInterceptDrags` + `Input.dispatchDragEvent` when source has `draggable="true"`), and per-keystroke typing. Three load-bearing invariants: (1) every action takes an explicit target — no stateful cursor cache (MV3 SW dormancy would wipe it silently); (2) `withTabInputLock(tabId, fn)` serialises mouse / drag / type / keyboard / scroll(at-pointer) per tab (CDP mouse state is global per attachment); (3) pre-click `elementFromPoint` hit-test fails with `target_obscured` if an overlay covers the target, bypass via `force: true`.

Every tool runs an implicit pre-flight (`ensureBridgeReady()` in `src/lib/browser-mcp/install-check.ts`) BEFORE acquiring an inflight slot (same load-bearing invariant as `predictedTooLong`). The pre-flight (1) auto-installs the native-messaging-host manifest for every detected browser (file write on POSIX, HKCU registry write + JSON file on Windows via `reg.exe`), and (2) probes the bridge's `<APP_DIR>/browser-mcp/bridge.json` discovery file + `/health` bearer-token-authed endpoint. On miss the dispatcher returns a structured `install_required` JSON text block with `isError: true` carrying a `load_unpacked_dir` path the user / model can act on. **Stable-dir materialization (`src/lib/browser-mcp/provision.ts`)**: because `npx`/`bunx` install into an ephemeral cache path that changes per version, the extension + bridge bundle are copied into a STABLE app-dir (`<APP_DIR>/browser-ext/` + `<APP_DIR>/browser-bridge/index.js`) on every launch — `provisionBrowserAssets()` runs fire-and-forget after `setupAndServe` (gated on the browse opt-in) AND lazily at the head of `ensureBridgeReady()`. The running `getPackageVersion()` is stamped into the materialized `manifest.json` in this one place on start, so a one-time "Load unpacked" of the stable dir survives upgrades and the version-mismatch `__reload__` picks up new code. The resolvers split: `bundledExtensionDir()`/`bundledBridgeBundlePath()` are the copy SOURCE (dist→src), while the runtime `extensionDir()`/`bridgeBundlePath()` prefer the stable copy (`GH_ROUTER_BROWSER_EXT_DIR` override wins for `extensionDir()`). Provisioning is signature-idempotent, single-flight, never throws; opt out with `GH_ROUTER_DISABLE_BROWSER_PROVISION=1` (also the lever the non-isolated gate test uses to stay hermetic). The extension's stable ID is derived deterministically from a fixed RSA pubkey in `src/browser-ext/manifest.json` `key` field via `computeExtensionIdFromKey()` (sha256-of-DER → first 16 bytes → hex-value-to-[a-p] map). The bridge bundle (materialized at `<APP_DIR>/browser-bridge/index.js`, bundled at `dist/browser-bridge/index.js`) is invoked by Chrome via a generated launcher shim that prefers `node` over `bun` (bun closes the bridge's binary stdin prematurely on SW dormancy). Navigation block is dual-layered: the bridge-side `preflightUrlPolicy()` in `src/lib/browser-mcp/policy.ts` checks `browser_open_tab` / `browser_navigate` URL args before forwarding, AND the extension's `webNavigation.onBeforeNavigate` listener cancels in-page-initiated nav (JS redirects, meta-refresh) to about:blank. `chrome://settings`, `chrome://extensions`, `chrome://flags`, password / management / policy pages, and `chrome-extension://*/options.html` blocked; `devtools://*` explicitly allowed. Opt-in audit log via `GH_ROUTER_LOG_BROWSER_MCP=1` (mirrors `GH_ROUTER_LOG_PEER_MCP` convention). Full architecture, install flow, per-OS NMH paths, MV3 SW dormancy + bridge grace-window handling, and the Playwright E2E harness in [`docs/browser-mcp-design.md`](docs/browser-mcp-design.md). E2E suite is gated on `GH_ROUTER_RUN_BROWSER_E2E=1` so it doesn't clash with the rest of `bun test` (some upstream tests mock `os.homedir()` which leaks into the bridge.json path resolution).

### Thinking-mode translation

Copilot rejects Anthropic's `thinking:{type:"enabled", budget_tokens:N}` shape on adaptive-thinking models with HTTP 400. The router translates to Copilot's `thinking:{type:"adaptive"}` + `output_config:{effort}` automatically when the resolved model declares `adaptive_thinking: true`. Bucket: `<2k → low`, `<8k → medium`, `<24k → high`, else `xhigh`. Clamps to `model.capabilities.supports.reasoning_effort` allowlist when present (lower-tier preference for ties). Client-supplied `output_config.effort` always wins. No-op when the model lacks `adaptive_thinking` (passthrough). Implemented in `src/routes/messages/handler.ts` (`translateThinking`).

### Semantic code search (folded into the `code` tool, ON by default)

ColBERT/PLAID semantic search is the **default mode of the unified `mcp__search__code` tool** (`mode: "semantic"`), backed by the router-managed [`colgrep`](https://github.com/lightonai/next-plaid) sidecar (`src/lib/colbert/`). There is no longer a standalone `semantic_search` tool. The routing lives in `src/lib/unified-code-search.ts` (`runUnifiedCodeSearch`), shared by both the MCP `code` tool and the worker subagent's internal `code_search` tool. **On by default** (opt OUT with `GH_ROUTER_DISABLE_SEMANTIC_SEARCH=1`): `start`/`claude`/`codex` fire-and-forget `provisionAndIndexColbert()` after `setupAndServe` (mirroring `provisionToolbelt()`), which SHA-pins-and-downloads three artifacts (the colgrep binary + ONNX Runtime CPU dylib + ColBERT INT8 model) into `~/.local/share/github-router/colbert/`, runs a **post-provision smoke test** (load-bearing — an invalid `ORT_DYLIB_PATH` makes colgrep silently fall through to its OWN unverified GitHub download; the smoke test scans stderr for colgrep's "not a loadable ONNX Runtime dylib; ignoring" warning and fails the handoff), then background-indexes the cwd if it's a git repo. **Transparent fallback (the merged-tool contract):** when colgrep isn't attemptable (artifacts absent / opted out) or the per-workspace index is `building`/`stale`/`unavailable`/`failed`, the `code` tool's semantic mode FALLS BACK to lexical BM25F and labels the response with a 3-valued `source` (`semantic` | `lexical` | `lexical-fallback`) so a degrade is never silent; semantic-ready rows additionally carry a ColBERT `score`. **Contract split (load-bearing):** `runSemanticSearch` in `src/lib/colbert/runner.ts` is UNCHANGED and still has its strict NO-LEXICAL-FALLBACK contract (honest `status`, never another engine); the fallback lives ONLY in the `unified-code-search.ts` tool layer. The availability predicate is the leaf `colbertSearchEnabled()` in `src/lib/colbert/` (single source of truth: `(NOT opted-out) AND colbertArtifactsPresent() AND colbertSmokeOk()`); `semanticSearchEnabled()` in `src/lib/mcp-capabilities.ts` delegates to it and no longer gates a tool's `tools/list` visibility — the `code` tool is ALWAYS listed (it always returns results via fallback), so the `search` group stays `{code, web}` regardless of colgrep availability (the **zero-regression guard**: on CI / sandboxes / pre-provision, colgrep is absent ⇒ semantic mode falls back to lexical, the surface is unchanged). The git **freshness verdict** (`src/lib/colbert/index-store.ts`) is the staleness correctness guard: a query is `fresh` only when `git rev-parse HEAD` matches the last index AND the tree isn't newly dirty — a branch switch or file deletion flips it to `stale` so a possibly-deleted-symbol hit is never labeled `ready`. colgrep is CLI-per-invocation (no daemon): per-search 30s timeout + stdout byte cap + tree-kill (POSIX process-group / Windows `taskkill /T`), an in-memory PID ledger reaped on SIGINT/SIGTERM/exit, and a boot-time sweep that reclassifies dead-`buildPid` `building` entries to `failed` (never killing a recycled PID from a prior boot). The managed binary is a native `.exe`, NOT a `.cmd` shim, so it bypasses `buildExecInvocation`'s cmd.exe path (which throws on `%`) via `runManagedExeCapture` (`src/lib/exec.ts`, `shell:false` everywhere — `%`/`&`-in-workspace-path safe). Re-pin SHAs with `bun run scripts/gen-colbert-manifest.ts`. Full design + adversarial-review record in [`docs/semantic-search.md`](docs/semantic-search.md) + [`docs/research/colbert-sidecar-design.md`](docs/research/colbert-sidecar-design.md).


### Web search

`/search` fulfils web-search tool calls via Copilot's MCP endpoint at `${copilotBaseUrl}/mcp` with the GitHub PAT directly (NOT the Copilot-exchanged token). Required `X-MCP-Toolsets: web_search` header; SSE-framed JSON-RPC; model-agnostic out-of-band fulfilment. PAT-bearing requests are gated by `COPILOT_HOST_ALLOWLIST`. Wire flow + Anthropic-shape `web_search_*` substitution in [`docs/web-search.md`](docs/web-search.md).

### Error format

Errors use Anthropic SDK format: `{type:"error",error:{type:"<category>",message:"..."}}`.
Upstream Anthropic-format errors from Copilot are forwarded as-is.
